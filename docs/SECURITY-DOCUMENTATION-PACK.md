## 1. ENCRYPTION ARCHITECTURE

### Overview

**Encryption Standard**: AES-256-GCM  
**Key Management**: AWS KMS with Envelope Encryption  
**Data Classification**: All PII and financial data encrypted at rest

### Architecture Diagram

```
┌─────────────────────────────────────────────────────────────┐
│                     AWS KMS (Customer Master Key)            │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  CMK (Customer Master Key)                            │  │
│  │  - Managed by AWS                                     │  │
│  │  - Automatic rotation every 90 days                   │  │
│  │  - Multi-region replication                           │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                            ↓ Encrypts
┌─────────────────────────────────────────────────────────────┐
│              Data Encryption Keys (DEK) - Per Tenant         │
│                                                              │
│  Tenant A: [Encrypted DEK stored in DB]                     │
│  Tenant B: [Encrypted DEK stored in DB]                     │
│  Tenant C: [Encrypted DEK stored in DB]                     │
└─────────────────────────────────────────────────────────────┘
                            ↓ Encrypts
┌─────────────────────────────────────────────────────────────┐
│                    Application Data                          │
│                                                              │
│  - Customer names                                            │
│  - Invoice amounts                                           │
│  - Payment details                                           │
│  - Financial records                                         │
└─────────────────────────────────────────────────────────────┘
```

### Encryption Flow

**Data Write (Encryption):**
1. Application requests DEK for tenant from database
2. KMS decrypts DEK using CMK
3. Application encrypts data using DEK (AES-256-GCM)
4. Encrypted data stored in database
5. DEK cleared from memory

**Data Read (Decryption):**
1. Application retrieves encrypted data from database
2. Application requests DEK for tenant
3. KMS decrypts DEK using CMK
4. Application decrypts data using DEK
5. Plaintext data returned to authorized user
6. DEK cleared from memory

### Encryption Scope

| Data Type | Encrypted at Rest | Encrypted in Transit | Key Type |
|-----------|-------------------|---------------------|----------|
| Customer Names | ✅ Yes | ✅ Yes (TLS 1.3) | Per-tenant DEK |
| Invoice Amounts | ✅ Yes | ✅ Yes (TLS 1.3) | Per-tenant DEK |
| Payment Details | ✅ Yes | ✅ Yes (TLS 1.3) | Per-tenant DEK |
| User Passwords | ✅ Yes (bcrypt) | ✅ Yes (TLS 1.3) | N/A (hashed) |
| JWT Tokens | ❌ No (signed) | ✅ Yes (TLS 1.3) | HMAC-SHA256 |
| Audit Logs | ❌ No (integrity) | ✅ Yes (TLS 1.3) | SHA-256 hash chain |

### Security Properties

- **Confidentiality**: AES-256-GCM provides 256-bit security
- **Integrity**: GCM mode provides authenticated encryption
- **Tenant Isolation**: Each tenant has unique DEK
- **Key Rotation**: Automatic CMK rotation every 90 days
- **Zero-Knowledge**: Plaintext keys never stored

## 2. KEY MANAGEMENT POLICY

### Key Lifecycle

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ Generate │ -> │  Active  │ -> │ Rotated  │ -> │ Archived │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
   Day 0          Day 1-90        Day 91         Day 365+
```

### Key Generation

**Customer Master Key (CMK):**
- Generated by AWS KMS
- 256-bit symmetric key
- Multi-region replication enabled
- Automatic backup

**Data Encryption Key (DEK):**
- Generated by AWS KMS GenerateDataKey API
- 256-bit symmetric key
- Encrypted by CMK before storage
- Unique per tenant

**Process:**
```bash
# Generate new tenant DEK
aws kms generate-data-key \
  --key-id alias/erp-cmk \
  --key-spec AES_256

# Returns:
# - Plaintext DEK (use immediately, then discard)
# - Encrypted DEK (store in database)
```

### Key Rotation

**Automatic Rotation (CMK):**
- Frequency: Every 90 days
- Method: AWS KMS automatic rotation
- Impact: Zero downtime (old keys retained for decryption)

**Manual Rotation (DEK):**
- Trigger: Security incident, compliance requirement, or scheduled
- Process:
  1. Generate new DEK
  2. Decrypt all data with old DEK
  3. Re-encrypt all data with new DEK
  4. Update database with new encrypted DEK
  5. Mark old DEK as rotated (retain for 30 days)

**Rotation Script:**
```bash
# Rotate tenant encryption key
npm run key:rotate -- --tenant-id=[TENANT_ID]

# Verify rotation
npm run key:verify -- --tenant-id=[TENANT_ID]
```

### Key Access Control

**Who Can Access Keys:**

| Role | CMK Access | DEK Access | Audit |
|------|-----------|-----------|-------|
| Application | Decrypt only | Decrypt only | ✅ All access logged |
| DevOps | ❌ No | ❌ No | N/A |
| Security Admin | Rotate, Disable | ❌ No | ✅ All access logged |
| AWS Root | Full | Full | ✅ All access logged |

**KMS Key Policy:**
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "Enable IAM User Permissions",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::ACCOUNT_ID:root"
      },
      "Action": "kms:*",
      "Resource": "*"
    },
    {
      "Sid": "Allow application to decrypt",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::ACCOUNT_ID:role/erp-api-role"
      },
      "Action": [
        "kms:Decrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "*"
    },
    {
      "Sid": "Allow CloudWatch Logs",
      "Effect": "Allow",
      "Principal": {
        "Service": "logs.amazonaws.com"
      },
      "Action": [
        "kms:Encrypt",
        "kms:Decrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "*"
    }
  ]
}
```

### Key Revocation

**Emergency Revocation:**
1. Disable CMK in AWS KMS console
2. All decryption operations fail immediately
3. Application enters read-only mode
4. Incident response team notified

**Procedure:**
```bash
# Disable key
aws kms disable-key --key-id alias/erp-cmk

# Schedule deletion (7-30 day waiting period)
aws kms schedule-key-deletion \
  --key-id alias/erp-cmk \
  --pending-window-in-days 30
```

### Key Backup

**CMK Backup:**
- Automatic by AWS KMS
- Multi-region replication
- Point-in-time recovery

**DEK Backup:**
- Stored in encrypted database backups
- Daily snapshots to S3
- 30-day retention
- Cross-region replication

## 4. DATA RETENTION POLICY

### Retention Periods

| Data Type | Retention Period | Reason | Deletion Method |
|-----------|------------------|--------|-----------------|
| Audit Logs | 1 year | Compliance (SOC 2) | Automated purge |
| Invoices | 7 years | Tax law | Soft delete |
| User Data | Account lifetime + 30 days | GDPR | Hard delete |
| Backups | 30 days | Disaster recovery | Automated expiration |
| Session Logs | 90 days | Security investigation | Automated purge |
| Error Logs | 30 days | Debugging | Automated purge |
| Quarantine Data | 90 days | Data quality | Automated purge |

### Data Deletion

**Soft Delete:**
- Record marked as deleted (`deleted_at` timestamp)
- Data remains in database
- Excluded from queries
- Recoverable within 30 days

**Hard Delete:**
- Record permanently removed from database
- Encryption keys destroyed
- Backups purged
- Not recoverable

**GDPR Right to Erasure:**
```bash
# User requests data deletion
npm run gdpr:delete-user -- --user-id=[USER_ID]

# Process:
# 1. Export user data (for compliance)
# 2. Anonymize audit logs (replace user_id with "DELETED_USER")
# 3. Hard delete user record
# 4. Hard delete all user-created data
# 5. Purge from backups
# 6. Send confirmation email
```

### Backup Retention

**Database Backups:**
- **Frequency**: Daily at 2:00 AM UTC
- **Retention**: 30 days
- **Storage**: AWS S3 with encryption
- **Replication**: Cross-region (us-east-1 → us-west-2)

**Point-in-Time Recovery:**
- **Enabled**: Yes
- **Retention**: 7 days
- **Granularity**: 5 minutes

## 5. AUDIT LOGGING ARCHITECTURE

### What is Logged

**All logged events:**
- User authentication (login, logout, failed attempts)
- Data access (read, write, delete, export)
- Permission changes
- Encryption key access
- Configuration changes
- API calls
- Security events (rate limit violations, suspicious activity)

### Log Format

```json
{
  "id": "uuid",
  "timestamp": "2024-02-07T12:34:56.789Z",
  "tenant_id": "tenant-uuid",
  "user_id": "user-uuid",
  "action": "READ",
  "resource_type": "invoices",
  "resource_id": "invoice-uuid",
  "ip_address": "192.168.1.100",
  "user_agent": "Mozilla/5.0...",
  "metadata": {
    "method": "GET",
    "url": "/api/invoices/123",
    "statusCode": 200
  },
  "previous_hash": "abc123...",
  "current_hash": "def456..."
}
```

### Tamper Detection

**Cryptographic Hash Chain:**
- Each log entry contains hash of previous entry
- Any modification breaks the chain
- Verification algorithm detects tampering

**Verification:**
```bash
# Verify audit log integrity
npm run audit:verify-chain

# Output:
# ✅ Audit log chain valid (10,523 entries verified)
# or
# ❌ Chain broken at entry 5,234
```

### Log Access

**Who Can Access:**
- ADMIN role: Full access
- MANAGER role: Own tenant only
- Security team: All tenants (with justification)
- Auditors: Read-only export

**Access Logging:**
- All audit log access is itself logged
- Creates audit trail of audit trail

## 6. DISASTER RECOVERY PLAN

### Recovery Objectives

- **RTO (Recovery Time Objective)**: 4 hours
- **RPO (Recovery Point Objective)**: 1 hour
- **Data Loss Tolerance**: Maximum 1 hour of transactions

### Disaster Scenarios

| Scenario | Likelihood | Impact | Recovery Time |
|----------|-----------|--------|---------------|
| Database failure | Low | High | 2 hours |
| Region outage | Very Low | Critical | 4 hours |
| Ransomware | Low | Critical | 6 hours |
| Data corruption | Medium | High | 3 hours |
| Accidental deletion | Medium | Medium | 1 hour |

### Recovery Procedures

**Database Failure:**
```bash
# 1. Promote read replica to primary
aws rds promote-read-replica \
  --db-instance-identifier erp-replica-1

# 2. Update application connection string
kubectl set env deployment/erp-api \
  DB_HOST=erp-replica-1.xxx.rds.amazonaws.com

# 3. Verify connectivity
npm run db:health-check

# 4. Resume operations
```

**Region Outage:**
```bash
# 1. Failover to secondary region
aws route53 change-resource-record-sets \
  --hosted-zone-id Z123456 \
  --change-batch file://failover.json

# 2. Restore database from cross-region backup
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier erp-prod-dr \
  --db-snapshot-identifier erp-backup-latest

# 3. Deploy application to DR region
kubectl apply -f k8s/dr-deployment.yaml

# 4. Verify all services operational
```
