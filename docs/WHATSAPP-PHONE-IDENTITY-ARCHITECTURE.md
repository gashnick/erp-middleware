# WhatsApp OTP + Phone-First Identity Implementation

## Overview

This implementation establishes a **phone-first identity architecture** for WhatsApp integration in a multi-tenant system. Phone numbers are now the primary identity for WhatsApp users, with strict enforcement rules preventing cross-tenant data leaks.

## Components Implemented

### 1. Database: Phone Number Column on Users

**File:** `src/database/migrations/system/1705000000016-AddPhoneNumberToUsers.ts`

```sql
-- Adds globally unique phone number column to public.users
ALTER TABLE public.users ADD COLUMN phone_number VARCHAR(20);

-- Global uniqueness constraint (E.164 format validation)
CREATE UNIQUE INDEX idx_users_phone_number_unique
ON public.users(phone_number)
WHERE phone_number IS NOT NULL AND deleted_at IS NULL;

-- Validate E.164 format: +<1-3 country code digits><1-14 local digits>
ALTER TABLE public.users
ADD CONSTRAINT valid_phone_number_format
CHECK (phone_number IS NULL OR phone_number ~ '^\+[1-9]\d{1,14}$');
```

**Why:** Phone is the first piece of identity WhatsApp provides. Must be globally unique across all tenants and users.

### 2. Entity: User Model Updated

**File:** `src/users/entities/user.entity.ts`

```typescript
@Column({ name: 'phone_number', length: 20, nullable: true, unique: true })
phoneNumber: string;
```

- Optional for email-only users
- Unique to prevent duplicate phone registrations
- Used by WhatsApp phone registry lookups

### 3. Existing Infrastructure (Already in place)

#### Public Schema Tables

**public.whatsapp_otp_requests** (system schema)

- Stores 6-digit OTP codes generated by authenticated users
- Tied to: `tenant_id` + `user_id` + `phone_number`
- `used_at` timestamp prevents replay attacks
- Expires after 15 minutes

**public.whatsapp_phone_registry** (system schema)

- Bridge table mapping phone → tenant + schema
- Written once at link-time
- Read on every inbound message to resolve tenant context
- Single source of truth for phone ownership

#### Tenant Schema Tables

**whatsapp_sessions** (tenant schema)

- One row per phone number per tenant
- `phone_number`: the inbound WhatsApp phone
- `user_id`: **MUST be non-NULL** after linking (enforced)
- `chat_session_id`: links to conversation history
- `context`: JSONB for conversation state

### 4. Enhanced WhatsApp Service

**File:** `src/whatsapp/whatsapp.service.ts`

#### New Strict Identity Enforcement

```typescript
// ✅ STRICT IDENTITY RULE in processTextMessage()
if (!session.userId) {
  this.logger.error(
    `SECURITY: Session exists for phone ${phone} but user_id is NULL. Blocking access.`,
  );
  // Send error message, do NOT route to LLM
  return;
}

// ✅ Double-check in routeToLlm()
if (!userId || !session.userId) {
  this.logger.error(`SECURITY VIOLATION: Attempted to route without user_id. Blocking.`);
  return;
}
```

#### Three-Phase Flow

**Phase 1: OTP Generation (App)**

```
User clicks "Get WhatsApp Code" in mobile app (authenticated)
  ↓
generateOtp(userId) called
  ↓
OTP inserted into public.whatsapp_otp_requests:
  - tenant_id: from user's current tenant
  - user_id: from authenticated request
  - expires_at: NOW + 15 minutes
  ↓
User receives code via notification
```

**Phase 2: OTP Validation (Webhook)**

```
User sends 6-digit code via WhatsApp
  ↓
handleInbound() webhook received (NO JWT, NO context)
  ↓
VALIDATE_AND_CONSUME_OTP_SQL (atomic):
  - Validates code exists, not expired, not used
  - Sets used_at = NOW (prevents replay)
  - RETURNS: user_id + tenant_id
  ↓
Query public.tenants → get schema_name
  ↓
Upsert whatsapp_sessions in tenant schema:
  - phone_number: from inbound message
  - user_id: ← POPULATED from OTP validation ✅
  - context: {}
  ↓
Register in public.whatsapp_phone_registry:
  - phone_number → tenant_id + schema_name + user_id
  ↓
Send success: "✅ Phone linked successfully!"
```

**Phase 3: Linked Phone Messaging**

```
User sends message: "What is our cash balance?"
  ↓
handleInbound() webhook (phone number only)
  ↓
LOOKUP_PHONE_SQL in public.whatsapp_phone_registry:
  - Returns: tenant_id + schema_name + user_id
  ↓
Query tenant schema whatsapp_sessions
  ↓
  ✅ STRICT CHECK: if session.user_id IS NULL → BLOCK
     (prevents orphaned sessions)
  ↓
routeToLlm() with verified identity:
  - user_id is guaranteed non-NULL
  - tenant_id is verified
  - schema_name is confirmed
  ↓
ChatService.handleMessage() (full LLM flow)
  ↓
Send response via Meta API
```

## Security Rules Enforced

### 🔒 Rule 1: Atomic OTP Consumption

- OTP validated and marked `used_at = NOW` in single transaction
- Prevents replay attacks
- Only valid if: `exists AND not expired AND not used`

### 🔒 Rule 2: Mandatory user_id in Session

```typescript
// In processTextMessage() after retrieving session
if (!session.userId) {
  // BLOCK: Cannot proceed without known user
  this.logger.error('SECURITY: Session missing user_id');
  return;
}
```

### 🔒 Rule 3: Phone Registry as Bridge Table

- Written once at OTP linking
- Source of truth for: phone → tenant_id + schema_name + user_id
- Every lookup goes through this table
- Prevents orphaned phones

### 🔒 Rule 4: No Operations Without Context

- All database operations in tenant schema use explicit `schema` parameter
- Tenant context set ONLY after phone registry lookup
- Prevents accidental cross-tenant queries

## Error Scenarios Handled

| Scenario                | Behavior                         | Message                             |
| ----------------------- | -------------------------------- | ----------------------------------- |
| Invalid/expired OTP     | Validation fails, reject         | "Invalid or expired code"           |
| Phone not registered    | Registry lookup fails            | "Complete WhatsApp setup in app"    |
| Session without user_id | Block access, log security error | "Please re-link your WhatsApp"      |
| Tenant not found        | Log error, reject                | "Linking failed — tenant not found" |
| Session doesn't exist   | Log warning, reject              | "Session error — please re-link"    |
